你现在是一个高级前端工程师、3D Web 开发工程师和产品设计师。请帮我从零开始开发一个网页网站项目，用来记录我和我女朋友一路以来旅游去过、玩过的地方。



项目名称暂定为：

Love Travel Globe / Our Memory Globe



项目核心目标：

我要做一个浪漫、可爱、互动感强的 3D 地球旅行相册网站。用户可以在 3D 地球上看到我和女朋友去过的国家、城市、景点，并且把照片存储在对应的国家、城市或具体地点里。



整个网站的视觉风格是：

\- 可爱

\- 粉色

\- 梦幻

\- 情侣旅行回忆风格

\- 柔和

\- 童话感

\- 玻璃拟态 UI

\- 圆角

\- 漂浮动画

\- 弹跳动画

\- 轻微发光效果



请注意：

不要一次性追求全球所有国家、城市、地标数据完整。这个项目应该先做一个可以运行的 MVP 版本，再逐步扩展。MVP 中可以先使用示例国家、城市、地标和占位 3D 模型，但代码结构必须支持后续继续扩展到更多国家、城市、地标、照片、音乐和动画。

技术要求：

1\. 使用 React + TypeScript + Vite 开发。

2\. 使用 CesiumJS 或 Three.js 实现 3D 地球视角。

3\. 地球处于粉色海洋背景中。

4\. 地球需要有持续性的动态效果，例如缓慢旋转、柔和光效、漂浮感。

5\. 项目结构必须清晰，不要把所有逻辑写在一个文件里。

6\. 代码需要拆分 components、data、types、hooks、services、lib、utils、styles、assets 等目录。

7\. 所有国家、城市、地标、照片、访问记录都必须使用数据驱动，不要硬编码在组件内部。

8\. 项目初始状态下，所有照片库都必须是空的。照片只能由用户后续通过网站上传功能添加。

9\. 每个去过的国家、城市、具体地点、地标和每一张照片都必须支持年月日时间标识。

用户账户、登录和云端存储系统：



重要架构改变：

本项目不是一个只在本地运行的网页，也不是只给我自己电脑使用的 local 网站。本项目需要做成一个真实可以上线的网站，所有用户都可以注册、登录并使用。



每个用户都需要拥有自己的账户。用户上传的照片、创建的旅行记录、去过的国家、城市、地点、相册、时间线、音乐设置等数据，都必须和该用户账户绑定。



不能让所有用户共用同一个照片库，也不能让一个用户看到或修改另一个用户的私人照片和旅行记录。



登录系统要求：

1\. 打开网站时，用户需要先进入登录页面。

2\. 未登录用户不能进入 3D 地球主页面。

3\. 用户可以注册新账户。

4\. 用户可以登录已有账户。

5\. 用户可以退出登录。

6\. 登录后进入自己的 3D 旅行地球。

7\. 每个用户看到的都是自己账户下的数据。

8\. 支持邮箱和密码登录。

9\. 后续可以扩展 Google 登录、Apple 登录等第三方登录。

10\. 登录页面也要保持粉色、梦幻、情侣旅行风格。



用户数据隔离要求：

1\. 每个用户有唯一 userId。

2\. 所有照片必须绑定 userId。

3\. 所有国家访问记录必须绑定 userId。

4\. 所有城市访问记录必须绑定 userId。

5\. 所有地标 / 地点访问记录必须绑定 userId。

6\. 所有相册数据必须绑定 userId。

7\. 所有背景音乐设置、筛选设置、偏好设置也要绑定 userId。

8\. 用户只能读取、创建、编辑、删除自己的数据。

9\. 不能读取、编辑、删除其他用户的数据。

10\. 云端数据库必须设置权限规则，不能只依赖前端隐藏。



推荐后端方案：

优先使用 Supabase，原因是：

1\. Supabase 支持用户认证 Auth。

2\. Supabase 使用 Postgres 数据库，适合存储国家、城市、地标、照片、访问记录这些关系型数据。

3\. Supabase Storage 可以存储用户上传的照片。

4\. Supabase 支持 Row Level Security，用于保证每个用户只能访问自己的数据。

5\. 代码结构要允许未来切换到 Firebase 或自建后端。



Firebase 只作为未来可能迁移或替代的方案，不在 MVP 阶段实现。MVP 阶段不要写 Firebase 相关代码。



正式版本不要使用 localStorage 作为主要存储：

1\. localStorage 只能保存少量 UI 偏好，例如侧边栏是否展开、主题偏好、最近选择的地图视角。

2\. 照片、照片元数据、旅行记录、用户数据必须存储在云端。

3\. IndexedDB 可以作为离线缓存或临时上传缓存，但不能作为正式唯一存储。

4\. 页面刷新、换设备、换浏览器后，用户登录同一个账户，仍然应该看到自己的照片和旅行记录。

2\. 数据库中保存照片的元数据。

5\. filePath 用于后续删除或更新照片。

基础地图数据和用户数据要分开：

1\. countries、cities、landmarks 是公共基础数据。

2\. 这些数据可以放在前端 data 文件中，或者未来放进公共数据库表。

3\. 用户上传的照片、访问状态、备注、旅行日期，必须放在用户自己的云端数据表中。

4\. 不要把用户照片直接写进 countries.ts、cities.ts、landmarks.ts。

5\. countries.ts、cities.ts、landmarks.ts 只存基础地理信息和默认地标信息。



数据读取流程：

1\. 用户打开网站。

2\. 如果未登录，跳转登录页。

3\. 用户登录成功。

4\. 根据 userId 从云端数据库读取该用户的：

&#x20;  - 去过的国家

&#x20;  - 去过的城市

&#x20;  - 访问记录

&#x20;  - 照片元数据

&#x20;  - 相册数据

&#x20;  - 用户设置

6\. 地球地图根据用户个人数据点亮国家、城市和地点。

7\. 相册只显示该用户自己的照片。



照片上传流程：

1\. 用户登录后才能上传照片。

2\. 用户选择照片文件。

3\. 用户填写照片标题、拍摄日期、时间、国家、城市、地点、备注和标签。

4\. 前端先把照片文件上传到云端 Storage。

5\. 上传成功后获得 fileUrl 和 filePath。

6\. 前端把照片元数据写入 photos 数据表。

7\. 同时更新对应国家、城市或地标的访问状态。

8\. 地图 UI 立即更新。

9\. 相册图标从空状态变成有照片状态。

10\. 播放上传成功动画和音效。



权限安全要求：

1\. 所有用户私有数据表都必须包含 userId。

2\. 数据库权限必须限制用户只能访问 userId 等于当前登录用户的数据。

3\. 云端照片存储也必须限制用户只能访问自己的照片文件。

4\. 不允许只在前端做权限判断。

5\. 管理端或调试模式也不能暴露其他用户私人照片。

6\. 未来分享功能必须单独设计 shared / public 权限，而不是默认公开。



部署上线要求：

1\. 项目需要可以部署成真实网站。

2\. 前端可以部署到 Vercel、Netlify 或其他静态网站托管平台。

3\. 后端数据使用 Supabase。

4\. 用户可以通过真实网址访问网站。

5\. 用户注册登录后，数据保存在云端。

6\. 用户下次再打开网站，登录同一账户后仍然能看到之前上传的照片、旅行记录和时间线。

7\. 用户换电脑、换浏览器、换设备后，只要登录同一账户，也应该看到自己的数据。



开发阶段要求：

第一阶段可以先完成登录页面 UI 和 Auth 接口结构。

第二阶段接入 Supabase Auth 登录。

第三阶段实现云端照片上传。

第四阶段实现云端数据库读写。

第五阶段实现权限规则。

第六阶段再部署上线。



请注意：

后端和云端服务选择：



本项目 MVP 阶段明确使用 Supabase，不使用 Firebase。



Supabase 负责：

1\. 用户注册和登录

2\. 用户身份验证

3\. 云端 Postgres 数据库

4\. 云端照片存储 Supabase Storage

5\. Row Level Security 用户数据隔离

6\. 用户设置和旅行记录存储



Firebase 只作为未来可能迁移或替代的方案，不在 MVP 阶段实现。

请不要同时实现 Supabase 和 Firebase，也不要写两套后端适配逻辑。MVP 阶段所有 Auth、Database、Storage 逻辑都基于 Supabase。



项目中不应该只使用 localStorage 或 IndexedDB 保存正式数据。localStorage 只能保存少量 UI 偏好，例如侧边栏是否展开、音量大小、主题设置等。



正式数据必须保存到 Supabase：

1\. 用户上传的照片

2\. 照片元数据

3\. 去过的国家

4\. 去过的城市

5\. 去过的地标

6\. 旅行时间记录

7\. 相册数据

8\. 用户设置
